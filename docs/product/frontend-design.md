# Frontend Design Direction

## Purpose

PatchPilot's frontend is an operations dashboard for maintainers, developers, and demos. Its job is to make the backend workflow visible: which issue triggered a task, what the agent did, where it failed, what tests ran, and which Pull Request was created.

The frontend is not the primary production trigger surface. GitHub issue comments remain the main task trigger, and GitHub Pull Requests remain the review surface. The dashboard may offer a manual task creation form for local demos and debugging, but it still goes through the backend queue.

## Design Philosophy

The interface should feel like an engineering control plane, not a chatbot, marketing page, or generic AI SaaS product. Operators should be able to scan task state quickly, inspect evidence, and jump back to GitHub when human review is needed.

Core principles:

- Prefer dense but readable operational views over large promotional sections.
- Show real task records, not abstract "AI progress" copy.
- Keep controls explicit and limited to safe operations.
- Make failures first-class: failed tests, cancelled tasks, queue errors, and agent exceptions should be easy to inspect.
- Make setup failures readable: backend/proxy connectivity problems should tell operators what to check.
- Preserve GitHub as the source of code review and merge decisions.

## Current Structure

The current frontend lives in `frontend/` and uses React, Vite, and TypeScript.

During local development, Vite proxies `/api` and `/health` to `PATCHPILOT_FRONTEND_BACKEND_URL` or `VITE_PATCHPILOT_BACKEND_URL` from the shell environment or repository root `.env`, defaulting to `http://127.0.0.1:8080`.

When the backend protects operator APIs with `PATCHPILOT_ADMIN_TOKEN`, the dashboard does not require a separate login flow. `App.tsx` shows a compact header control that reports whether this browser has a saved admin token, lets an operator save or replace the local token, and lets the operator clear it when rotating credentials or testing unauthenticated behavior. For trusted localhost-only development, `PATCHPILOT_DASHBOARD_ADMIN_TOKEN_BOOTSTRAP_ENABLED=true` lets `GET /api/dashboard/bootstrap` return the configured admin token before protected API loading starts; the dashboard stores that token only when no browser token already exists. If a protected API call returns `Admin token is required`, the same save-and-retry path appears inline inside the alert. Saved tokens live in browser `localStorage` as `patchpilot.adminToken`; the API helper reads that value for later requests and sends it as `X-PatchPilot-Admin-Token`.

The page coordinator is `frontend/src/App.tsx`. It loads backend data, owns selected-task state, applies status filters, repository owner/name filters, language/build-system adapter filters, created time range filters, task-list sort, and search state, coordinates manual trigger evaluation, manual creation, cancel, and retry actions, and runs local repository preflight diagnostics through `POST /api/repository-preflight`. It loads `/health` and then tries the dashboard bootstrap endpoint before the protected dashboard API batch, so the connectivity panel can distinguish "backend/proxy unavailable" from "backend is up but protected APIs need an admin token." The operator setup checklist is deliberately derived from the same loaded data instead of a new API, so it acts as a first-screen summary of the detailed panels below it.
It also loads `GET /api/demo/evidence-bundle`, `GET /api/demo/session-snapshot`, `GET /api/demo/session-archives`, `GET /api/demo/script`, `GET /api/demo/readiness`, `GET /api/demo/smoke-checklist`, `GET /api/github/credential-readiness`, `GET /api/github/repository-access-readiness`, `GET /api/language-adapters`, `GET /api/language-adapters/fixtures`, `GET /api/language-adapters/runtime-readiness`, `GET /api/evaluation/cases`, `GET /api/evaluation/summary`, `GET /api/evaluation/case-readiness`, `GET /api/evaluation/run-preview`, `GET /api/evaluation/fixture-baseline-runs`, `GET /api/rejected-triggers`, `GET /api/rejected-triggers/summary`, `GET /api/trigger-quarantines`, `GET /api/admin-audit-events`, and `GET /api/github/webhook-deliveries` so demo evidence, a full demo session snapshot, recent archived session reports, the read-only demo script, readiness, final smoke-check steps, GitHub token acceptance, selected repository access, supported repository shapes, fixed verification commands, detection signals, demo fixture paths, current fixture pass/fail status, current adapter executable availability, a copyable adapter readiness report, a read-only evaluation case catalog, readiness summary, fixture-readiness report, run preview, and recent archived fixture baseline runs, selected accepted-trigger evidence, recent rejected task triggers, rejected-trigger abuse patterns, active trigger quarantines, recent protected admin mutations, recent GitHub delivery outcomes, and redelivery guidance are visible in the dashboard instead of only in source code, terminal smoke output, curl output, or GitHub's delivery page. It calls `GET /api/demo/runbook` only when an operator clicks `Copy runbook`, `POST /api/demo/session-report` only when an operator clicks `Copy session report` with prepared command and archived outcome context, `POST /api/demo/session-report/download` only when an operator clicks `Download session report` with prepared command and archived outcome context, `POST /api/demo/handoff-package` only when an operator clicks `Copy handoff package` with prepared command and archived outcome context, `POST /api/demo/handoff-package/download` only when an operator clicks `Download handoff package` with prepared command and archived outcome context, `GET /api/demo/session-archives/{archiveId}/report/download` only when an operator downloads one archived report, `POST /api/demo/session-archives` only when an operator clicks `Archive session`, `POST /api/evaluation/fixture-baseline` only when an operator clicks `Run fixture baseline`, `POST /api/evaluation/fixture-baseline-runs` only when an operator clicks `Run and archive fixture baseline`, `GET /api/evaluation/fixture-baseline-runs/{runId}/report/download` only when an operator downloads one archived baseline run report, and `POST /api/github/webhook-diagnostics/evaluate-payload` only when an operator submits pasted webhook diagnostic fields, so Markdown reports, handoff packages, archive records, fixture baseline execution, archived fixture baseline evidence, and payload diagnostics are created or retrieved from current operator input without adding another task-triggering path. GET session-report endpoints remain compatible for API clients that do not have browser-local command or outcome context. The evaluation case catalog panel is read-only except for the explicit fixture baseline actions, which run adapter-selected verification commands against checked-in local fixtures only; they do not create tasks, call the model, mutate Git, or write to GitHub. The trigger-decision panel reuses the selected task timeline and rejected-trigger data to compare accepted and rejected decisions in one place. The admin audit trail panel renders the protected mutation stream, supports server-backed action/operator/resource filtering, and copies a Markdown evidence report for the visible rows, while the rejected-trigger panel filters quarantine-related safety audit rows from that stream and posts manual trigger quarantine create and release actions to the same protected quarantine API so operators can block abusive trigger users or repositories and recover false positives without leaving the dashboard.
It also loads `GET /api/task-queue/worker-health` with the queue summary and queue item APIs so operators can see whether the backend worker poller is not started, polling, idle, active, or most recently errored without reading process logs.
It calls `POST /api/demo/launch-command` only when an operator submits the launch command composer, turning structured repository, issue, operation, target path, and replacement text fields into a copyable `/agent fix` comment and reusable launch-preflight input without creating tasks or mutating GitHub. Successful command compositions are stored only in browser `localStorage` as `patchpilot.demoLaunchCommandHistory`, keeping the five most recent unique commands available for copy, composer refill, launch-preflight reuse, session-report handoff evidence, final launch-package evidence, and post-launch tracking during repeated demo practice. It calls `POST /api/demo/launch-preflight` only when an operator submits the launch preflight form, combining current demo readiness with a read-only GitHub issue-comment trigger dry run for the exact `/agent fix` comment that will be posted. After a launch preflight result exists, the panel can copy a Markdown launch package that combines the exact GitHub issue URL, comment, readiness and trigger-gate evidence, browser-local prepared commands, and next actions without creating another backend record. The demo launch tracker uses the same browser-local prepared commands and correlates them with already-loaded webhook delivery diagnostics, task records, and Pull Request URLs so operators can see whether a posted launch is still waiting for webhook delivery, running, failed, or ready for Pull Request review, then copy a Markdown outcome report for a specific launch without creating another backend record. Operators can also archive recent outcome reports in browser `localStorage` as `patchpilot.demoLaunchOutcomeArchive`, reopen archived task and Pull Request links, copy archived reports again, and clear that local archive. The demo session snapshot reads the same browser-local outcome archive so copied, downloaded, and archived session reports can include both the prepared launch commands and the observed launch outcomes.

Selected-task detail uses the `/tasks/{taskId}` frontend route and `GET /api/tasks/{taskId}/detail`, an aggregate read-model endpoint that returns the task audit summary, latest queue item, queue history, accepted-trigger intent audit, latest generated diff, timeline events, test runs, tool calls, model calls, and failed-task diagnosis together. For accepted tasks, the trigger intent audit shows the safety-gate result, issue-context load status, and model trigger-classification outcome as structured fields before the operator scans raw timeline events. For failed and cancelled tasks, the page also loads `GET /api/tasks/{taskId}/retry-preflight` so retry eligibility, stable category, sanitized reason, and next operator action are visible before an operator queues another attempt. The task summary includes selected adapter metadata and a nullable detection reason, which the detail evidence strip renders next to the verification command so operators can see why the task chose that language/build path. If the generated-diff risk gate rejects a patch, the task appears as `PENDING_REVIEW`, the same evidence strip shows a `Risk gate BLOCKED` marker, the generated-diff panel shows the exact patch to inspect before approval, and the tool-call list shows the concrete rejection reason. Operators can cancel pending-review tasks or approve the existing generated diff with an approver selected from the configured review-approval allowlist plus a reason. Approval requeues the task to resume the same workspace after the risk gate instead of asking the model to produce a new patch, and approved tasks show the stored approval audit metadata in detail. If no review approvers are configured, the approval form is disabled and explains the missing configuration. Retry remains hidden for pending-review tasks because retry is reserved for failed or cancelled fresh runs. Failed and cancelled tasks show a `Retry preflight` section; retry remains enabled for categories that can be safely requeued and is disabled as `Retry blocked` when GitHub permissions or repository support must be fixed first. Retry requires an operator-entered reason before the button is enabled, and retried tasks show that reason with the previous status, failure reason, and retry time in retry lineage. Failed tasks with a stored status-comment URL label that GitHub link as `Failure feedback`, because the issue-facing comment includes the failure category, next action, and sanitized reason. Failed tasks also show a `Failure diagnosis` section with the same stable category, next action, and sanitized reason before raw evidence. Pending-review tasks label the same link as `Review feedback`; other tasks keep `Status Comment`. Legacy `?taskId=` links still select the same task. Status, search, repository owner/name filters, adapter filters, created time filters, and non-default sort state are stored as URL query parameters, so `/tasks/{taskId}?status=PENDING_REVIEW&query=maven&repositoryOwner=bingqin2&repositoryName=PatchPilot&language=node&buildSystem=npm&sort=createdAtAsc&createdAfter=2026-06-20T01:00:00Z&createdBefore=2026-06-21T01:00:00Z` restores the selected task and task-list view together. The clear-filter action removes `status`, `query`, `repositoryOwner`, `repositoryName`, `language`, `buildSystem`, `createdAfter`, and `createdBefore`, preserving sort, the selected task route, unrelated query parameters, and hash fragments. Markdown task reports use `GET /api/tasks/{taskId}/report` so operators can copy a compact diagnostic summary, trigger intent audit, failure diagnosis, retry reason, and generated diff without manually assembling API responses. This keeps the dashboard detail panel focused on the selected task while preserving narrower backend endpoints for curl-based debugging.

## Current UX Model

The first screen is the working dashboard:

- Demo readiness summarizes credentials, safety policy state, demo target policy alignment, adapter fixture verification, queue health, worker heartbeat readiness, and recent Pull Request evidence before an operator starts a live smoke run.
- The demo evidence bundle gives one status, count set, latest evidence links, and next-action list for live-demo readiness.
- The demo session snapshot gives one session id, share summary, prepared launch commands, archived launch outcomes, checklist, health contract, current evidence, script context, copyable and downloadable Markdown session reports, copyable and downloadable Markdown handoff packages, and recent archived reports for pre-run confirmation or post-run handoff.
- The demo script turns that evidence into a guided path with concrete commands, success criteria, troubleshooting panels, and a read-only contract before the operator touches GitHub.
- The smoke checklist turns readiness, webhook delivery, task execution, and Pull Request evidence into ordered final checks before the operator posts the live `/agent fix` comment.
- The demo launch command composer turns structured fields into a validated `/agent fix` comment, exposes the target GitHub issue, supports copying the command, stores the five most recent commands in local browser history, can refill the composer from that history, and can fill the launch preflight form directly.
- The demo launch preflight lets the operator paste the exact issue comment planned for GitHub, confirms whether current readiness plus issue-comment task-creation gates allow it, and copies either the raw preflight evidence or a full launch package with issue URL, comment, prepared command context, and next actions before the live comment is posted.
- The demo launch tracker turns those prepared commands into post-launch rows by matching them to recent webhook, task, and Pull Request evidence so the operator can tell whether to wait, inspect the task, troubleshoot webhook delivery, or review the Pull Request, then copy a compact outcome report for handoff or save it to the local outcome archive.
- Metrics summarize task health.
- A compact refresh status tells operators when top-level dashboard data is still loading, and the title area shows when the dashboard last refreshed successfully.
- Operational summaries highlight failure causes, model usage, and latency without requiring terminal inspection. Failure causes show stable categories such as `VERIFICATION_FAILED` and `GITHUB_OPERATION_FAILED` with the next action returned by the backend metrics API.
- A manual task form can evaluate a local demo/debug `/agent fix` trigger before creation, switch the dry-run source between manual API and GitHub issue-comment gate behavior, copy a Markdown evidence report for the dry-run decision, then enqueue the task through the same backend task and queue path used by webhook-created tasks.
- The task list supports URL-backed status filters with scoped count badges, repository owner/name filters, language/build-system adapter filters, created time range filters, backend-backed search over task history, newest/oldest sorting, one-click filter reset, loaded-versus-total counts, and incremental loading.
- Status count badges come from `GET /api/tasks/status-counts`. They follow search, repository, adapter, and created-time filters, but ignore the active status, sort, limit, and offset so operators can see the distribution before switching status tabs.
- Metrics and operational summary panels use the same search, repository, adapter, and created-time scope as the task list, so a filtered Node/npm view does not show global Maven/Python task totals.
- Selecting a task updates the `/tasks/{taskId}` route while preserving active filter query parameters, loads aggregate task detail in one request, and reveals a copyable task link, copyable Markdown report, selected language adapter metadata with detection reason, accepted-trigger intent audit when present, failed-task diagnosis when present, queue state, queue history, execution evidence summary, timeline events, verification output, tool-call records, model-call records, and GitHub links.
- Generated-diff risk-gate failures are visible before scrolling through raw records. The generated-diff preview shows the exact patch before approval, with `GeneratedDiffRiskGate` still available in tool-call evidence for exact rejection details.
- Pending-review tasks expose an explicit approve-review form that records an approver and reason before resuming the already-generated workspace.
- Queue visibility shows whether work is pending, delayed, running, failed, or cancelled, with failed/delayed/running health hints before the row list. The same panel shows worker heartbeat state, readiness status, last poll age, operator action, poll count, claimed count, the latest claimed task id, and the latest worker error so operators can distinguish queued work from an inactive, stale, or failing poller.
- Rejected-trigger visibility shows why a `/agent fix` comment did not create a task when the safety gate, allowlists, rate limiter, rejected-trigger quarantine, or model classifier intentionally refused it. Each row shows a category badge, and the panel can filter by category through the URL-backed `rejectedCategory` parameter so operators can isolate vague requests, dangerous instructions, unauthorized users, blocked repositories, rate limits, abuse quarantine, and model-classifier refusals without parsing reason text. Rows also expose retry eligibility before an operator acts: direct retry remains enabled only for safe actionability/model-classification cases, while dangerous, unauthorized, rate-limited, quarantined, unsupported, unknown, or already-retried rows show `Retry blocked` with the next action. The panel also summarizes the recent rejection window by category, source, top trigger users, and top repositories; category summary rows are clickable and reuse the same filter. Active trigger quarantines are shown above the audit rows so operators can see the current enforced safety state before reading individual refusals, manually add a temporary trigger-user or repository block, inspect the evidence behind a selected quarantine, and release a false-positive quarantine. Recent operator safety audit rows are shown in the same panel so manual create/release actions are traceable by operator, reason, target, and time.
- Admin audit visibility shows protected operator mutations with server-backed action, operator, resource type, resource id, and scope-key filters. The panel can copy the current filtered rows as Markdown evidence for a demo handoff, incident note, or pull-request review context.
- Webhook delivery visibility shows whether recent GitHub deliveries reached PatchPilot, why they were ignored or rejected, and which task id was created when a delivery succeeded. Pasted payload diagnostics let an operator validate a raw GitHub delivery payload and optional signature without creating task, queue, rejection, delivery, or GitHub comment records.
- Trigger decision visibility shows why the selected `/agent` comment was accepted and what recent comments were refused instead, using the selected task's `TRIGGER_ACCEPTED` timeline event plus recent rejected-trigger rows. When a rejected row has a GitHub refusal comment URL, the quick comparison view includes a `Refusal comment` link so an operator can open the issue-facing explanation without leaving the trigger-decision context.
- Manual trigger evaluation visibility shows whether a proposed local `/agent fix` would create a task or be blocked, which source was evaluated, which gate made that decision, whether issue context was loaded for model classification, and the next operator action. The copy action exports the evaluated repository, issue, trigger user, command, status, blocked reason/category, per-gate decisions, issue-context state, and next action as Markdown for demos, issue comments, or review notes. It is read-only and does not create tasks, queue items, rejected-trigger audit rows, GitHub comments, webhook delivery diagnostics, or rate-limit records.
- Trigger intent visibility in task detail turns the accepted-trigger timeline evidence into safety, issue-context, and model-decision fields so copied reports and per-task investigation show why execution proceeded.
- Configuration visibility shows backend `/health` status, the active provider, model, workspace root, queue policy, worker heartbeat stale threshold, trigger rate-limit thresholds, rejected-trigger quarantine thresholds, generated-diff risk-gate state, protected path pattern count, repository-preflight allowed roots, whether required secrets are configured, and clear health hints for missing secrets or weak optional settings without exposing secret values.
- Supported-adapter visibility shows each supported language/build system, verification command, detection signals, and demo fixture path. Adapter readiness visibility summarizes the same data with fixture pass rate, language coverage, allowlisted commands, runtime executable availability, fixture failures, and copyable Markdown evidence. Fixture verification visibility shows whether each demo fixture still maps to the expected adapter and command. If any adapter API fails, the readiness and detailed panels show local warnings while the rest of the dashboard can still load.
- Evaluation case catalog visibility shows the first benchmark map for Java/Maven, Node/npm, Python/pytest, Go, unsafe-trigger rejection, and vague-trigger rejection cases. It gives operators readiness status, fixture-readiness status, next action, expected commands, changed files, missing fixture files, adapter drift, success criteria, safety expectations, a run preview report with known gaps, and recent archived fixture baseline runs. Operators can copy static benchmark evidence without executing cases, or explicitly run and archive the checked-in fixture baseline when they need executable local evidence.
- Repository preflight visibility lets an operator enter an allowed local path such as `docs/demo-repositories/java-maven`, see the configured allowed roots, run the real adapter registry, and see whether that repository shape would be supported before creating a task. After a result exists, the operator can copy a Markdown preflight report with status, repository path, selected adapter, verification command, detection reason, operator action, allowed roots, and supported adapter options for unsupported repositories.
- If the readiness API fails, the demo readiness panel shows a local warning while the rest of the dashboard can still load.
- Cancel and retry are available only for task states where those actions make sense; failed and cancelled tasks show retry preflight before queuing another attempt, require an operator reason before retrying, and retried tasks keep source failure context plus the retry reason in the detail panel.

This keeps the UI focused on operator questions:

- What is running right now?
- Is the queue worker alive, idle, active, or failing?
- Is the queue worker fresh enough for a live smoke run?
- What failed and why?
- Did tests pass?
- Which language adapter and verification command ran?
- Why was that language adapter selected?
- Which repository shapes are supported right now?
- Are all adapter fixtures still ready for a multi-language demo?
- Do the checked-in demo fixtures still detect as the expected adapters?
- Can I copy evidence that a specific local repository path is supported before posting `/agent fix`?
- Why did this `/agent fix` comment not create a task?
- Why was this `/agent fix` comment accepted while similar recent comments were rejected?
- What did the agent call?
- How many model tokens were used?
- Where is the GitHub issue or Pull Request?
- Did GitHub's latest webhook delivery reach PatchPilot and, if not, was it a signature, payload, safety, duplicate, or active-task issue?
- Is there enough live-demo evidence to proceed, or should setup be fixed first?
- Can I share a URL that reopens the same task detail?
- Can I copy a compact report that explains the selected task state?

## Final Target

The long-term frontend target is a complete maintainer dashboard for safe issue-to-PR automation.

Target capabilities:

- Search and filter complete task history.
- Inspect task timeline, generated diff, model calls, tool calls, command runs, and test records.
- Track queue health, retries, cancellation, and worker recovery.
- Surface cost, latency, success rate, failure rate, and test pass rate.
- Link every important UI object back to GitHub.
- Provide safe operator controls such as cancel, retry, generated-diff review approval, and configuration inspection.
- Make safety decisions observable by showing rejected trigger audit records, active trigger quarantines, operator safety audit records, and recent rejection-pattern summaries in the same control plane as accepted tasks.
- Prevent unsafe retry actions by showing rejected-trigger retry eligibility and blocked reasons before creating any task.
- Prevent blind task retries by showing task retry preflight, requiring an operator retry reason, and disabling retry when setup or repository support work is required first.
- Make accepted and rejected trigger decisions comparable from the selected task view before an operator drills into full audit or quarantine controls.
- Make accepted trigger decisions globally auditable so operators can confirm which recent `/agent fix` comments were allowed, which gates passed, and which task each decision created.
- Let operators dry-run a manual `/agent fix` trigger before task creation, using the same backend gates without mutating task, queue, GitHub, rejected-trigger, or rate-limit state, and copy the dry-run gate evidence for review or demo handoff.
- Let operators compose a controlled demo `/agent fix` comment from structured inputs, copy it, and apply it to launch preflight without creating a task or mutating GitHub.
- Let operators reuse recently composed demo `/agent fix` comments from browser-local history during repeated smoke runs without creating tasks, mutating GitHub, or depending on terminal notes.
- Let operators run a final demo launch preflight for the exact GitHub issue comment that will be posted, combining readiness and trigger-gate evidence before the live webhook path starts.
- Let operators copy a final demo launch package after preflight so the issue URL, exact comment, gate evidence, prepared command history, and next actions are preserved before posting on GitHub.
- Let operators track the posted demo launch from prepared command to webhook delivery, task state, and Pull Request review link without manually comparing multiple panels.
- Let operators copy a Markdown outcome report for a tracked demo launch so the exact command, webhook result, task result, Pull Request URL, and next action survive after the live run.
- Let operators keep a browser-local archive of recent demo launch outcome reports so repeated smoke runs remain reviewable after refresh without adding backend state.
- Let operators copy or download a Markdown handoff package that summarizes session evidence, prepared command counts, archived outcome counts, recent task and Pull Request evidence, next actions, and the embedded session report.
- Make accepted-trigger intent available inside task detail and copied reports so task-level investigation explains why execution was allowed before showing what the agent did.
- Show the supported adapter matrix, fixture paths, and fixture pass/fail verification for demo planning and troubleshooting.
- Show the evaluation case catalog, fixture-readiness report, and archived fixture baseline runs as the first evaluation-harness surface, proving which language and safety scenarios PatchPilot intends to measure, whether the checked-in fixtures still support that claim, and what the latest local fixture executions produced before automated benchmark runs are added.
- Support manual demo trigger evaluation and task creation while keeping creation on the durable backend queue.
- Support demo readiness by making a full task lifecycle understandable without reading logs.
- Support a read-only demo script that gives operators the next action and verification command without adding another task-triggering path.
- Support final smoke-check decisions by pairing every live-demo step with current evidence and a next action.
- Support demo session archiving so operators can keep a recent record of the exact Markdown report used during a live demo, with database-backed local profiles preserving those records across backend restarts.
- Include browser-local prepared demo launch commands and archived demo launch outcomes in copied, downloaded, and archived session reports so handoff records show which controlled GitHub issue comments were prepared and what happened after they were posted.

## Non-Goals

The frontend should not:

- Replace GitHub issue comments as the primary production task trigger.
- Auto-merge Pull Requests.
- Replace GitHub's review UI.
- Hide task execution behind a chat-only interface.
- Become a marketing landing page.
- Execute arbitrary commands from the browser.

## Success Criteria

The frontend is successful when a maintainer can open it during a live demo or local test run and answer these questions without terminal access:

- Which task was created from which GitHub issue?
- What is its current status?
- What did the agent do?
- Which tools and model calls ran?
- Did verification pass?
- If it failed, what was the failure reason?
- If it succeeded, where is the Pull Request?
